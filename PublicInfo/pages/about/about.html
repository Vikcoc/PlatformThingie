Welcome!

To a hopefully microservices app


This page is ment to serve as the entry point into what hopefully will be the example of me being able to take a microservices app from A to Z.


Why?

Mostly to aquire and prove my expertise as a software developer.


What and how:

I plan the theme of the project to be some version of an inventory management platform. I think it lends itself well to being broken into multiple small services:

  • One service to define the inventory. It is supposed to manage information about manufacturer, sque, colors and other properties of the items
  • One service to actually keep track of the items
  • One service to plan the aquisition of items
  • One service to plan the dispozal of items

If these would be a monolit, they would still be separate parts of the same application. Especially since different people would have different responsabilities; A regular employee can take, for example, a laptop home, but may not be the one allowed to decide what laptops should be bought. Forcing everything in a monolit, in my opinion, encourages the abstractisation of aspects that, although seemengly related, managing the state of an item, have vastly different requirements.


For example:

Changing the laptop of an employee may only require to note that the assignment has been made. On the other hand, buying that said laptop may need documentation for the need, documentation for the available options from vendors, budged approvals, and finally an order.

Dealing with vastly different workflows in a monolit can result in one of likely two options:

  1. Completely separate workflows, which is ripe for separating into a different service
  2. Accomodating the workflows into a parent workflow, which would be a hurdle, and would introduce the possibility for changes to one workflow to disrupt the other

I want to only use .NET APIs for the web server functionality as an exercise to make well defined modules that can be used between them. One example would be a module to standardise the asyncronous communication between the microservices.

As developing for frontend is easier with Npm and Node there will also be a Node project.


On the user side I want to have as vanilla of a website as possible, to not be framework dependent. HTML and CSS pages that will be served with short scripts to add functionality. I will take heavy inspiration from Material and probably even use web components provided by them.

Taking inspiration from Material I will probably also make components with Lit, to reuse functionality like on the backend.

I think Lit is a fair choice as far as being framework independent, as using Rollup seems to compile all the necessary javascript into a single file. Hopefully it will work forever like the original Space Jam website.
